1. Missing: approved_quantity on lease_items
When an approver changes the quantity of an item, you can't overwrite quantity_requested — that's the original ask and must be preserved for audit. Add a new column:

approved_quantity INT DEFAULT NULL

NULL means "same as requested". If approver changes it, write to approved_quantity only. The issued quantity then comes from approved_quantity ?? quantity_requested.
2. Missing: stock count view
Three pages (landing, approver detail, center issue) all need "how many of SKU X are available at Center Y". Rather than repeating a GROUP BY query everywhere, add a MySQL view:

CREATE VIEW sku_stock_by_center AS SELECT sku_id, center_id, COUNT(*) as available_count FROM assets WHERE status='available' GROUP BY sku_id, center_id;

FastAPI queries this view — simple and fast.
3. Missing: transfer completion page
Transfer is initiated at the source center. But who marks it complete? The destination center admin needs a UI to "receive" the asset — which updates assets.center_id to the new center and sets asset_transfers.status='completed'.

Add a small "Incoming Transfers" tab to /center/inventory or /center/dashboard showing in_transit assets coming to this center, with a "Mark Received" button.
4. Missing: center_id on audit_log
The center dashboard activity feed needs to show actions scoped to that center. Right now audit_log has no center_id — you'd need to join through entity_id which is messy.

Add: center_id CHAR(36) to audit_log with a FK to centers. FastAPI writes this when logging actions that have a center context.
5. Token number generation
token_number is VARCHAR(20) UNIQUE in lease_requests, but the generation logic isn't defined anywhere. FastAPI should generate it as:

MDF- + 6 random alphanumeric uppercase chars

Generate, attempt INSERT, catch duplicate key error, regenerate. In practice with 36^6 = 2.1B combinations for ~1000 requests, collision probability is negligible but the retry logic must still exist.
6. Partial returns not tracked
A lease_request with 2 items could have one returned and one still out. The current schema handles this (each lease_item has its own returned_at) but lease_requests.status needs a new value:

Add 'partially_returned' to the status ENUM in lease_requests. FastAPI sets this when some but not all lease_items are returned.
