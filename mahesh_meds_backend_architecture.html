Status translation layer (API ↔ DB)
The DB and API use different status names. Translation happens on every read and write.
Asset status
API: under_repair↔DB: repair all others pass through unchanged
Lease request status
API: active↔DB: issued or partially_returned API: closed↔DB: returned
resolve_lease_db_status() — server-authoritative fulfillment
Called on every create/update of a lease request. Derives status from line-item asset assignments.
  • If current status is rejected, returned, or partially_returned → return as-is (terminal states)
  • If all line items have an asset_id assigned → status = issued
  • If status was issued but some items now have no asset_id → status = approved (de-assignment)
  • Otherwise → keep current status unchanged
Lease request lifecycle
pending → approver → approved → issue devices → issued / active → return → returned / closed

Authentication pattern (no JWT — mobile-based)
  • No session middleware or JWT in main.py — auth is fully frontend-managed
  • Frontend passes X-Actor-Mobile and X-Actor-Role headers on every request
  • Backend uses these to look up the User and write actor context into audit_log
  • otp_sessions table exists in DB but has no API endpoints yet — OTP flow is pending
Audit log write — automatic on every mutation
  • Every POST/PATCH/DELETE calls write_audit_log() after committing the main change
  • Actor context comes from X-Actor-Mobile header → looked up in users table
  • actor_mobile and actor_role are always embedded in new_value JSON
  • Section classifier routes each log row to: admin, approver, requests, or centers
  • Some GET requests also write audit events (list views, token lookups)
  • JSON serializer handles datetime, Decimal, UUID types safely before storing
Issue-context endpoint — cross-center availability planning
  • Takes a token + center_id; finds all unassigned line items (asset_id = NULL)
  • Queries available asset counts per SKU grouped by every center
  • For each line item returns: local_available_count (this center) + source_centers (other centers with stock)
  • Drives the "Issue Device" page — lets center staff see whether to issue locally or initiate a transfer
